# 测试用例编写

为已有代码补充测试用例：先确定范围、探测项目已有的测试框架，分析预期行为列出用例清单并经用户确认，再落地编写，最后实际运行验证其有效性（语法对了不算过，要真的测到点上）。

## 为什么这样设计

跳过"列清单确认"直接写测试，容易测了不该测的点、漏掉真正的边界情况，且用户事后才发现方向不对。跳过"实际运行验证"直接交付，可能连测试本身的语法/断言都有问题却没人发现。默认不改业务代码，是因为修复 bug 属于难以一眼回溯的操作，应该报告给用户由其决定，不能在"补测试"的名义下顺手夹带。

## 测试范围的确定

- 用户点名了文件 / 模块 / 功能 → 以此为范围。
- 用户没点名 → 默认以 `git diff` 与 `git diff --staged` 的未提交改动为范围；若也没有改动，询问用户要为哪部分代码编写测试。

## 探测项目测试框架与约定

查找现有测试目录/文件命名规则（如 `tests/`、`__tests__/`、`*_test.go`、`*.spec.ts`）、测试框架（pytest / unittest / jest / vitest / go test / cargo test 等）与运行命令（`package.json` scripts、Makefile、CI 配置）。

若仓库没有任何测试基线，**不要擅自引入新框架**：向用户说明现状，询问期望使用的框架与运行方式，等待确认后再继续。

## 列出用例清单（先确认，再写代码）

阅读目标代码，列出需要覆盖的场景：正常路径、边界值、异常/错误处理、已知的隐藏约束。把用例清单（场景描述，先不写代码）展示给用户确认，避免遗漏或测错重点。

## 编写测试用例

按确认后的清单逐个编写测试，遵循项目现有的命名、断言风格与目录结构。只写测试代码，不顺手修改被测的业务代码；编写过程中若发现业务代码本身有 bug，先报告给用户，不要自行"顺手"修复。

## 运行验证

实际执行项目的测试命令，确认新增用例都能跑起来。若新用例失败，判断是测试本身写错还是确实发现了被测代码的 bug，分别处理：前者修正测试，后者向用户报告而不擅自改动业务代码（除非用户要求）。完整报告测试结果（通过/失败数量），不要只展示部分输出就声称"通过"。

## 中止条件

- 项目无可识别的测试框架，且用户未给出期望方案。
- 测试范围内代码行为模糊、无法判断"预期"是什么，需要用户先澄清需求。
