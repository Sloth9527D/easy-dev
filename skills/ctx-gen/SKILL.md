# 项目上下文知识生成器

扫描当前仓库，提炼出四份主题文档存放在指定目录，并把它们通过 `@` 引用挂进 `CLAUDE.md`，使后续所有 Claude 会话都能自动获得这份上下文。**若文档已存在，直接读入上下文，跳过分析和生成步骤。**

## 为什么这样设计

Claude Code 会在会话启动时自动加载 `CLAUDE.md`，而 `CLAUDE.md` 里的 `@path/to/file` 引用会把对应文件的内容也拉进来。所以生成四份独立的主题 md、再把引用挂进 `CLAUDE.md`，就实现了"一次生成、永久生效"——比把所有内容塞进单个 `CLAUDE.md` 更清晰，也方便按主题单独更新。

先检查再生成，避免对已有上下文做无谓的重复分析，也允许用户手工维护这些文档而不担心被覆盖。

## 执行步骤

### 0. 确定输出目录并检查已有文档（首先执行）

**确定目标目录**：从用户请求中提取（如"生成到 docs/"），否则默认 `.claude/context/`。

**检查四个文件是否存在**（并行 Read，文件不存在时 Read 会返回错误，以此判断）：
- `<目标目录>/architecture.md`
- `<目标目录>/conventions.md`
- `<目标目录>/tech-stack.md`
- `<目标目录>/domain.md`

根据结果分三种情况处理：

**全部已存在** → 将四份文档内容读入当前会话上下文，然后告知用户"已加载现有上下文文档，共 N 行"，**直接结束，跳过步骤 1–4**。这是最常见的日常使用路径。

**部分已存在** → 读入已有的文件（加入上下文），记录缺失的文件列表，仅对缺失文件执行步骤 1–3 中对应的分析和生成。在步骤 5 汇报时说明哪些是读取的、哪些是新生成的。

**全部不存在** → 继续执行完整的步骤 1–4。

### 1. 并行探索仓库基础信息

同时收集以下内容，为后续深度阅读做准备：

- **目录结构**：用 Glob 列出顶层目录和主要子目录，识别项目骨架
- **技术栈线索文件**：查找 `package.json`、`requirements.txt`、`pyproject.toml`、`CMakeLists.txt`、`go.mod`、`Cargo.toml`、`*.csproj`、`Makefile`、`setup.py` 等
- **现有文档**：读取 `CLAUDE.md`（已有内容不重复覆盖）和 `README.md`/`README.rst`（若存在）
- **CI 配置**：`.github/workflows/`、`Jenkinsfile`、`.gitlab-ci.yml` 等，能反映构建/测试命令

### 2. 深度阅读关键源文件

根据上一步识别的技术栈，有选择地阅读：

- **主入口**：`main.py`、`index.ts`、`main.go`、`src/main.rs`、`cmd/*/main.go` 等（≤2 个）
- **核心模块**：按目录结构判断哪几个文件最核心，各读 50–100 行（共 2–4 个文件）
- **测试文件**：随机抽 1–2 个，了解测试框架和命名风格
- **配置文件**：任何描述工具链或运行参数的配置

目标不是读完所有文件，而是**读够了能回答「新来的工程师想改一个功能应该去哪里动手」这个问题**。

### 3. 生成四份主题文档

创建 `.claude/context/` 目录（若不存在），写入以下四个文件。内容要**具体且有代码依据**——每个结论都应能从你实际读到的文件里找到出处，不要写"可能有"之类的模糊措辞。

#### `.claude/context/architecture.md`

```
# 架构概览

## 目录结构
<带注释的目录树，说明每个主要目录的职责>

## 核心模块
| 模块/文件 | 职责（一句话） |
|-----------|----------------|
| ...       | ...            |

## 依赖方向
<用箭头或文字描述模块间谁依赖谁，特别是核心抽象层>

## 扩展点
新增功能通常在哪里动手（具体到文件/函数/注册表）
```

#### `.claude/context/conventions.md`

```
# 代码约定

## 命名
<文件名、变量、函数、类的命名风格，各给 1–2 个真实代码里的例子>

## 错误处理
<如何表达失败：抛异常、返回码、Result 类型、错误回调——用代码片段说明>

## 注释风格
<是否写注释，写什么类型（类型注释、docstring、行内注释）>

## 测试约定
<测试文件在哪、命名规则、使用的框架、mock 策略>

## 提交格式
<若仓库有约定，写出格式（如 Conventional Commits），否则标注"未发现明确约定">
```

#### `.claude/context/tech-stack.md`

```
# 技术栈与常用命令

## 语言与框架
<主要语言、关键依赖库及版本（从配置文件里读）>

## 常用命令
| 用途     | 命令                      |
|----------|---------------------------|
| 安装依赖 | ...                       |
| 构建     | ...                       |
| 运行     | ...                       |
| 测试     | ...                       |
| 格式化   | ...（若有）               |

## 工具链
<包管理器、编译器、linter、formatter 等，版本若可知则写上>

## 环境要求
<运行时版本要求、必须存在的环境变量或外部依赖>
```

#### `.claude/context/domain.md`

```
# 领域背景

## 项目解决的问题
<2–3 句话，说清楚这个项目是什么、给谁用、解决什么痛点>

## 核心概念与术语
| 术语 | 含义 |
|------|------|
| ... | ... |

## 关键流程
<主要操作路径（如：用户触发 X → 经过 Y 模块 → 输出 Z），用步骤列表或流程描述>

## 设计决策与已知约束
<为什么这样架构（而不是另一种方式），有什么硬性约束或历史包袱>
```

**质量检查**：写完每份文档后，对照自己读过的源文件，确认：
- 每个"事实性描述"都有代码依据，不是猜测
- 没有"可能"、"大概"、"似乎"之类的模糊词（有把握才写，没把握就不写）
- 每份文档在 100–150 行以内；超出说明写了太多废话，裁剪

### 4. 更新 CLAUDE.md

在 `CLAUDE.md` 末尾追加下面这段（若已存在相同引用则跳过）：

```markdown

## 项目上下文（自动生成）

@.claude/context/architecture.md
@.claude/context/conventions.md
@.claude/context/tech-stack.md
@.claude/context/domain.md
```

若 `CLAUDE.md` 不存在，创建一个只含上述引用的最简版本。

### 5. 汇报结果

输出一段简洁汇总，根据实际执行路径有所不同：

- **全部已加载**：说明读取了哪些文件、各文件的大致内容（1 句），当前会话已可直接利用这些上下文
- **部分生成**：区分"已读取"和"新生成"两组，各给 1 句摘要
- **全部新生成**：列出路径和每份文档的 1–2 句摘要，提示用户运行 `/reload-plugins` 或重启会话后 `@` 引用会自动生效

## 关于后续维护

生成的文档不是"写一次完事"。当仓库发生大的架构变化（重命名模块、引入新框架、调整目录）时，建议重新运行此技能更新对应文档，或手动编辑 `.claude/context/` 下的具体文件。修改范围最小化：只更新变了的那份，其余保持不动。
