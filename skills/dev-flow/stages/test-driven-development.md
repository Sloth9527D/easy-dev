# 卡点 A（阶段 4 内）：测试驱动开发循环

业务代码必须由一个先写好、且先看着它失败的测试驱动出来，而不是先写实现再补测试。这是一条**文字约束**——靠自我检查执行，没有自动化 hook 拦截，所以每一步都要诚实地按顺序走，不能图快跳步。

## 为什么这样设计

先写实现再补测试，测试很容易被写成"验证我刚写的代码是这样做的"，而不是"验证需求被满足"——会掩盖逻辑漏洞，也测不出真正的边界情况。强制先看到红，再让它变绿，能确保测试本身是有效的（至少曾经失败过），且实现只做到刚好让测试通过，不多不少。

## 适用判断

- 原子任务的验证步骤涉及业务逻辑，而不是纯文档/配置/脚本编排改动。
- 本仓库（或当前 worktree）有可运行的测试命令。如果阶段 2 已确认"无测试基线"，本卡点不适用，直接按 plan.md 给出的验证步骤手动验证即可，不要为了套用 TDD 而临时发明一个测试框架。

## 执行步骤（RED → GREEN → REFACTOR）

### 1. RED：先写一个会失败的测试

根据原子任务描述的预期行为，写一个新的测试用例，断言这个行为成立。**此时不要触碰业务代码**。

### 2. 运行测试，确认它确实失败

实际执行测试命令，亲眼看到这个新测试失败（不是猜测"应该会失败"）。如果它意外通过了，说明测试没测到点上，回到步骤 1 重写。

### 3. GREEN：写最精简的实现

只写刚好能让这个测试通过的代码，不要顺手实现 spec 之外的分支或做额外抽象。再次运行测试，确认通过，且没有破坏其他已有测试。

### 4. REFACTOR：必要时重构

测试全绿之后，如果实现里有重复或命名不清的地方，可以重构，但每次重构后都要重新跑一遍测试确认仍然全绿。如果当前实现已经足够清晰，跳过这一步，不要为了"重构"而重构。

### 5. 提交

确认测试全绿后再提交这一个原子任务的改动（提交粒度与 plan.md 中的任务粒度一致）。

## 自查规则（替代自动化拦截）

每次准备编辑业务代码文件前，先问自己："对应的失败测试已经写好并跑过了吗？" 如果答案是否——也就是已经抢先写了业务代码——立即停下来，**撤销刚写的业务代码改动**（如刚创建的文件用 Write 清空/删除，刚编辑的文件用 Edit 还原到改动前的内容），回到步骤 1 先补测试。不要带着"先写的实现"继续往后走，也不要事后补一个刚好覆盖该实现的测试敷衍过去。
