# 阶段 1：需求澄清与头脑风暴

在动手写代码前，先把用户的想法从"粗糙"打磨成"可执行的设计方案"。不直接写代码，而是通过提问把隐藏的假设、备选方案和验收标准都摆到桌面上，最后落成一份双方都认可的 spec 文档。

## 为什么这样设计

粗糙的需求里藏着大量未声明的决策（用什么落地形式、要不要兼容旧行为、边界条件怎么处理）。如果跳过澄清直接写代码，这些决策会被隐式地"猜"掉，往往要返工。先分章节把方案摆出来让用户确认，比写完代码再讨论便宜得多。

## 适用判断

- 用户描述的是目标或痛点，而不是具体到文件/函数的改法。
- 任务有架构选择空间（落地位置、是否引入新依赖、影响范围）。
- 这不是单纯的 bug 修复或已经给出精确规格的小改动——那种情况不需要走本流程，直接做即可。

## 执行步骤

### 1. 先复述理解，暴露假设

用一两句话复述你理解的目标，列出你看到的关键假设（例如"我假设这个功能只面向当前插件，不涉及其他仓库"）。让用户在这一步纠正方向，比后面纠正代码便宜。

### 2. 苏格拉底式提问，逐步收窄

不要一次性抛出所有问题。优先用 `AskUserQuestion` 围绕真正影响架构的决策点提问，每次聚焦 1-4 个关键问题，例如：
- 落地形式（新增 skill / 改现有命令 / 纯脚本）
- 触发条件与边界（什么时候生效、什么时候不生效）
- 失败时的行为（报错、回退、静默跳过）
- 是否需要兼容已有调用方

如果用户的回答暴露了新的分支决策，继续追问，不要急于收尾。

### 3. 分章节展示设计方案

把澄清后的方案拆成简短的章节逐一展示给用户确认（例如："目标""涉及范围""不做什么""落地形式""关键决策与理由"），而不是一次甩出一篇长文。每个章节给用户机会喊停或改方向。

### 4. 保存 spec 文档

全部章节确认后，把最终方案写成 Markdown 保存到 `.claude/dev-flow/<slug>/spec.md`（`<slug>` 用英文短横线命名，从功能名提炼，如 `auto-changelog`）。spec 至少包含：

```markdown
# <功能名> 设计方案

## 目标
<这个改动要解决什么问题，一两句话>

## 范围与不做什么
<明确边界，避免范围蔓延>

## 设计决策
<关键决策与理由，对应澄清阶段的问答结论>

## 验收标准
<怎样算做完——可观察、可验证的标准>
```

写入前用 Read 检查目标目录是否已有未完成的 spec，避免覆盖用户尚在跟进的另一个功能。

### 5. 交接下一阶段

保存完成后，明确告知用户 spec 路径，并说明下一步是阶段 2（隔离工作区）——但**不要自动继续**，等待用户对 spec 本身的最终确认（"可以开始了"之类的明确许可）后才推进，因为 spec 仍可能需要修改。
