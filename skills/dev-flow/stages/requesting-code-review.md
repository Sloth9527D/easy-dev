# 卡点 B（阶段 4 内）：任务间审查

每完成一个原子任务，就停下来对照 spec 和 plan 检查一遍改动，而不是攒到最后一次性审查。问题分级处理：致命问题直接挡住后续任务，轻微问题记录下来但不阻塞。

> 这是工作流**内部**的卡点，对照 spec/plan 检查。若想对任意改动做不依赖工作流的通用代码检视，用独立的 `code-inspect` 技能。

## 为什么这样设计

等所有任务做完才审查，发现致命问题时返工成本最高——可能要推翻好几个已经"完成"的任务。每个原子任务后立刻审查，能把问题挡在它产生的那一刻，后续任务也不会站在错误的地基上继续叠。

## 适用判断

阶段 4 中每个子智能体完成一个原子任务后自动触发；也可由用户随时主动要求。

## 执行步骤

### 1. 收集本次改动范围

```powershell
git diff HEAD -- <本次任务涉及的文件路径>
```

只看这一个原子任务对应的改动，不要把还没轮到的任务范围也卷进来评审。

### 2. 对照 spec 与 plan 检查

读取 `.claude/dev-flow/<slug>/spec.md` 和 `plan.md` 中对应任务的小节，检查：

- **是否符合 spec**：改动是否实现了任务描述的行为？有没有偷偷做了 spec 之外的事，或漏做了 spec 要求的一部分？
- **代码质量**：命名、是否有明显重复逻辑、错误处理是否合理、是否引入了不必要的复杂度。
- **验证是否真的跑过**：plan.md 里这条任务写的验证步骤，是否有证据表明真的执行过（测试输出、命令结果），而不是声称"应该没问题"。

### 3. 按严重程度分类报告

| 级别 | 含义 | 处理方式 |
|------|------|----------|
| Critical | 偏离 spec、破坏现有功能、引入安全问题、测试造假/被跳过 | **熔断**：停止派发后续任务，要求先修复 |
| Major | 实现可用但有明显质量问题（重复代码、缺边界处理） | 记录，建议在当前任务内顺手修复，不强制熔断 |
| Minor | 风格、命名、可读性等细节 | 记录到汇报里，不阻塞 |

### 4. 熔断处理（仅 Critical）

发现 Critical 问题时，明确告知用户：哪个任务、什么问题、为什么判定为致命。**不要自动尝试修复**——交回给阶段 4 决定是重新派发同一任务还是由用户介入。在问题解决并重新通过本审查之前，不进入计划中的下一条任务。

### 5. 通过

没有 Critical 问题时，简短汇报本次任务的审查结论（包含记录到的 Major/Minor 问题，如果有），放行进入下一条原子任务。
