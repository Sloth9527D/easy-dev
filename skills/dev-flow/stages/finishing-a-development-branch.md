# 阶段 5：完成并清理分支

功能分支上的所有原子任务都完成之后，做一次全局验证，再让用户决定这条分支的归宿，最后清理掉临时工作区。这是整条工作流的收尾，确保不会留下孤儿 worktree 或半成品分支。

## 为什么这样设计

逐任务验证只能保证局部正确，合并前补一次全局测试能抓住任务之间的交互问题。把"怎么处理这条分支"交给用户决定（而不是默认直接合并），是因为合并、删除分支这类操作影响共享状态、难以撤销，必须由用户拍板。

## 适用判断

阶段 4 确认计划下全部任务已 `completed` 之后。

## 执行步骤

### 1. 全局测试验证

在 worktree 内运行本仓库的全局测试命令（若阶段 2 已确认本仓库无测试基线，这一步改为：浏览 `git diff` 全量改动，确认与 spec 验收标准逐项对应）。

```powershell
PowerShell -Command "cd $worktreePath; <测试命令若存在>"
```

如果全局验证发现问题，停下来报告，不要在这一步自行决定如何修复——回退到阶段 4 的审查/派发处理具体问题，修好后再回到本步骤重跑。

### 2. 汇总改动

展示这条分支相对主分支的提交列表（`git log main..feature/<slug> --oneline`）和改动统计（`git diff main...feature/<slug> --stat`），让用户对改动范围有整体认识。

### 3. 让用户选择收尾方式

用 `AskUserQuestion` 提供以下选项，不要默认替用户选：

- **合并到主分支**：在主工作区执行合并（`git merge feature/<slug>` 或按用户习惯的合并方式），合并前确认主分支工作区无未提交的冲突改动。
- **创建 PR**：推送分支并引导用户创建 Pull Request（推送前确认远程仓库地址是用户预期的）。
- **保留分支，先不处理**：仅清理 worktree 目录，分支本身保留，供用户后续手动处理。
- **放弃这次修改**：删除分支与 worktree——这是破坏性操作，执行前必须再次明确确认，说清楚会丢失哪些提交。

### 4. 执行所选操作

按用户的选择执行对应 git 操作。合并、推送、删除分支这几类操作都是难以撤销或影响共享状态的——执行前用一句话说明将要做什么，执行后报告结果；任何报错（冲突、权限、远程拒绝）都如实展示给用户，不自动重试或强行覆盖。

### 5. 清理 worktree

无论用户选了哪种收尾方式（除非用户明确要求保留 worktree 以便继续看代码），都清理掉临时工作区：

```powershell
git worktree remove $worktreePath
```

若 `worktree remove` 因为有未提交改动而失败，停下来告知用户，不要加 `--force` 强行删除未确认的改动。

### 6. 收尾汇报

告知用户：最终选择的处理方式、分支/worktree 当前状态、本次 `.claude/dev-flow/<slug>/` 下的 spec 与 plan 文档是否需要保留作为历史记录（默认保留，除非用户要求清理）。
