# 阶段 3：制定微级执行计划

把已批准的 spec 翻译成一份可以无脑执行的原子任务清单——每条任务足够小、足够具体，执行者（哪怕是一个全新的、不了解上下文的子智能体）也能照着做完并自行验证。

## 为什么这样设计

任务粒度太大（"实现功能 X"）会让执行阶段又变回一次性大改动，难以中途检查、难以回滚到某个干净点。拆成 2~5 分钟的原子任务，每个任务都能独立验证、独立提交，配合阶段 4 逐项派发执行时，出问题能精确定位到哪一条任务，而不是整片改动。

## 适用判断

紧接在阶段 2 完成之后；前提是 `.claude/dev-flow/<slug>/spec.md` 已存在且已获批准。

## 执行步骤

### 1. 读 spec，梳理改动面

读取 `.claude/dev-flow/<slug>/spec.md`，结合 Glob/Grep 确认 spec 中提到的文件/模块在当前代码里的真实路径（不要凭空假设路径，spec 里描述的是意图，实际路径要在代码里核实）。

### 2. 拆分原子任务

按"自然的提交边界"拆分，常见顺序：先底层（数据结构/脚本/接口），再上层（调用方/UI/文档）。每条任务必须满足：

- **足够小**：一个人或一个子智能体能在 2~5 分钟内完成并验证，不依赖"顺手再改一下别的地方"。
- **自包含**：写清楚要改哪个文件、改成什么样（给出关键代码片段或明确的修改描述，不能只写"实现 XX 逻辑"这种空泛指令）。
- **可验证**：给出具体的验证步骤（运行什么命令、检查什么输出、断言什么行为），不是"看起来对就行"。
- **有序但松耦合**：标注任务间的依赖关系（哪些必须在哪些之后），没有依赖关系的任务可并行处理。

如果某一步的实现涉及业务代码而非纯配置/文档，在该任务描述里提示"遵循 TDD：先写失败测试再写实现"（见 `stages/test-driven-development.md`），除非这个仓库当前确实没有可用的测试基线（沿用阶段 2 的结论，不要前后矛盾）。

### 3. 落成计划文档

写入 `.claude/dev-flow/<slug>/plan.md`：

```markdown
# <功能名> 执行计划

## 任务 1：<一句话标题>
- **文件**：<路径>
- **改动**：<具体要写的代码逻辑或改动描述>
- **依赖**：<无 / 任务 N>
- **验证**：<命令或检查步骤>

## 任务 2：...
```

### 4. 同步到任务跟踪

用 `TaskCreate` 把每条原子任务登记为一个 task（subject 用任务标题，description 引用 plan.md 中的对应小节），方便阶段 4 按 task 状态派发与追踪进度。

### 5. 请求批准

把任务清单的概览（标题 + 顺序/依赖关系）展示给用户，等待"开始执行"之类的明确指令后，才进入阶段 4——不要自动开始写代码。
