# 阶段 4：自动化派发与执行

把阶段 3 产出的原子任务逐条派发给全新的子智能体执行，而不是在当前对话里一口气写完所有改动。每个子智能体只带着任务描述和必要的文件路径开工，做完后经过两道评审卡点再标记完成。

## 为什么这样设计

每个原子任务用一个全新的子智能体执行，能保证它严格按照任务描述本身行事，不会被当前对话里积累的、与本任务无关的上下文带偏。两道卡点（先对 spec、再对代码质量）把"做的是不是这件事"和"做得好不好"分开检查，问题能更快定位到具体维度。

## 适用判断

紧接在阶段 3 给出任务清单、用户明确下达执行指令之后。

## 执行步骤

### 1. 确认起点

用 `TaskList` 查看阶段 3 登记的任务及其依赖关系，找出当前可以开始的任务（无未完成的前置依赖）。

### 2. 询问执行模式

如果用户没有指明，先确认：是**连续自主执行**（适合任务清单已经很清楚、用户希望无人值守跑完）还是**每完成一批任务留一个人工检查点**（适合任务有不确定性、用户想中途看一眼）。没有强烈信号时，默认在每个 Critical 级别问题出现时停下来汇报，其余情况连续执行。

### 3. 逐任务派发

对每个就绪任务，使用 `Agent` 工具派发一个全新的子智能体，prompt 中必须包含：

- 这条原子任务在 plan.md 中的完整描述（文件路径、要写的代码逻辑、验证步骤）——子智能体看不到当前对话，prompt 必须自包含。
- 明确指示：如果任务涉及业务代码改动且本仓库有可用测试基线，要遵循 TDD（先写失败测试再实现，详见 `stages/test-driven-development.md` 的 RED-GREEN-REFACTOR 与自查规则）。
- 明确指示：完成后自行运行验证步骤，并在返回结果中报告验证是否通过。

把对应 `TaskUpdate` 为 `in_progress`，派发的子智能体跑完后再看结果。

### 4. 任务完成后过两道卡点

子智能体返回结果后：

1. 按 `stages/requesting-code-review.md` 对这条任务的改动做审查。
2. 若结果是 Critical → 不标记任务完成，按代码审查阶段的熔断处理：停下来向用户汇报，等待修复指示后再重新派发同一任务（可派发新的子智能体修复，或交还用户处理）。
3. 若通过 → 用 `TaskUpdate` 标记该任务 `completed`，继续下一个就绪任务。

### 5. 人工检查点（若选择该模式）

按用户约定的节奏（例如每完成 3 条任务，或每个里程碑），暂停下来汇总已完成任务和发现的 Major/Minor 问题，等待用户确认后再继续派发。

### 6. 全部任务完成

当 `TaskList` 中本次计划下的任务全部 `completed`，告知用户进入阶段 5（收尾清理）。
