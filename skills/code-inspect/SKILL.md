# 独立代码检视

对一段改动做系统的代码审查，按严重程度分级报告问题。这是一个**通用、独立**的检视技能——不依赖任何工作流，可随时对工作区改动、某几个文件或整条分支发起。默认只产出报告、不动代码；只有用户明确要求修复时，才把改动应用到工作区。

## 为什么这样设计

代码检视的价值在于"发现并分级"，而不是悄悄改掉——把判断权留给作者本人，能避免审查者误改意图、也方便作者从问题中学习。分级（而非一锅端）让作者能先处理致命问题，把风格细节留到有空时。默认不改代码，是因为修改代码属于难以一眼回溯的操作，应由作者确认后再动。

## 检视范围的确定

根据用户的话确定要看什么，按以下优先级：

1. **用户点名了文件/目录** → 只看这些（用 Read 读全文，必要时 Grep 找相关调用方）。
2. **用户说"这个分支" / 给了分支名** → `git diff <base>...<branch>`，base 默认 `main`。
3. **没指明** → 默认看当前未提交的改动：先 `git status` 看全貌，再 `git diff`（未暂存）和 `git diff --staged`（已暂存）。

读改动时不要只看 diff 的几行——对每处改动，用 Read 把所在函数/文件的上下文也读进来，否则会漏掉"改动本身没错但破坏了周边假设"这类问题。

## 检视维度

逐处改动对照以下维度，**有依据才写，没把握的不硬凑**：

- **正确性**：逻辑是否符合意图？边界条件（空值、越界、并发、错误路径）有没有处理？有没有引入回归？
- **可读性 / 可维护性**：命名是否表意？是否有重复逻辑可抽取？是否有不必要的复杂度？是否偏离了本仓库已有的写法（参照周边代码与 `.claude/context/conventions.md`（若存在））？
- **性能**：有没有明显的低效（重复计算、N+1、在循环里做 IO、不必要的全量扫描）？仅在确有依据时提，不臆测。
- **安全**：是否有注入、路径穿越、硬编码密钥、未校验的外部输入、危险的命令拼接？
- **约定一致性**：提交粒度、错误处理风格、注释密度是否与本仓库一致（本仓库提交遵循 Conventional Commits，见 `commands/commit.md`）。

## 严重程度分级

| 级别 | 含义 | 建议 |
|------|------|------|
| Critical | 会导致功能错误、数据损坏、安全漏洞、破坏现有行为 | 必须修复后再提交 |
| Major | 明显的质量/可维护性问题（重复逻辑、缺边界处理、不当抽象） | 建议本次修复 |
| Minor | 风格、命名、可读性细节 | 可择机处理 |
| Nit | 纯偏好、无关紧要 | 仅供参考 |

## 报告格式

用中文输出，按严重程度从高到低组织。每条问题包含：位置（`文件:行号`，可点击）、是什么问题、为什么是问题、建议怎么改。例如：

```
## Critical
- `script/install_tool/cli.py:42` — 未捕获网络异常，下载失败会直接崩溃中断后续工具安装。建议包一层 try/except 并归入结尾的失败汇总。

## Major
- ...

## Minor / Nit
- ...
```

如果某个维度没发现问题，不必硬写"无问题"占位；整体没发现任何问题时，明确说"未发现 Critical/Major 问题"，并简述检视覆盖了哪些范围，让用户知道审查的边界。

## 修复（仅在用户明确要求时）

默认**只报告、不改代码**。只有当用户明确说"帮我修""把这些改掉""按你的建议改"之类时，才用 Edit 应用修复，并遵循：

- 优先只修 Critical 和用户点名的项，不顺手改无关代码。
- 每修一处，简述改了什么、对应报告里的哪一条。
- 改完提示用户复核 `git diff`，并由用户决定是否提交（不替用户提交——提交是外向、需确认的操作）。
- 如果修复涉及业务逻辑且本仓库有测试基线，提醒用户跑一遍测试确认未引入回归。
